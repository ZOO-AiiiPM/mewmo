# Mewmo 测试 SOP

## 决策原则

每次先按改动影响面选择验证，识别被改变的用户行为、模块、运行边界、数据边界和环境。只运行能证明这些变化正确的专项测试；与改动无关的套件不进入本次清单。耗时是范围合理后的结果，不是先把测试分成“快”和“慢”。

改动跨越多个边界时合并对应证据：纯逻辑跑聚焦单元测试；API 改动加验认证、ownership、输入、持久化和错误分支；UI 行为加浏览器交互；UI 视觉真实切换深色和浅色并验收关键尺寸；缓存和性能改动验证命中、刷新、失败回退、隔离、mutation 一致性和请求数量。

## 可执行入口

- `pnpm test:unit` 运行无 Web、数据库、Redis、固定账号或外网依赖的自包含测试。
- `pnpm test:integration` 使用一次性容器、独立端口、唯一账号和本地 fixture 运行 API 集成测试，并在结束后清理。
- `pnpm test:theme` 拒绝新增的主题相关硬编码白色或黑色；例外只能写入 `tooling/theme-color-allowlist.json`，并精确到文件和代码片段、附理由。
- `pnpm test` 等于 `pnpm test:unit`，保证干净环境可重复。
- `pnpm verify` 运行 lint、自包含测试、主题策略和生产构建。它是生产最低本地闸门，不代替本次改动要求的集成或浏览器证据。

## UI 主题验收

普通应用 UI 使用 `--ink`、`--ink-soft`、`--canvas`、surface、line、hover 和 selected 等语义变量。不能用 `#fff`、`white`、`text-white` 等固定前景色让深色界面“看起来正常”却在浅色中不可读。品牌图形、图片遮罩和明确反色控件可使用有理由的集中白名单。

自动扫描只是早期防线。UI 改动还要在浏览器真实切换深浅主题，检查正文、辅助文字、图标、输入和 placeholder、边框、hover、选中、禁用、危险状态、弹窗、Popover、Toast、编辑器和外部内容。构建成功不是视觉证据。

## 断言更新

断言表达已批准的行为契约，依据来自用户要求、已确认 spec、项目规则、API schema、数据约束或可复现缺陷，不能只依据当前代码长相。失败后先分类：需求未变是实现回归，修实现；需求已确认改变才更新断言；环境漂移修环境或 harness；过度绑定源码文本时升级为行为断言。不得只为让测试变绿而削弱断言。

## 环境边界

本地负责聚焦调试、真实服务依赖和浏览器验收；GitHub CI 从干净 checkout 运行项目脚本；Vercel 证明生产产物构建和部署。CI 的 `localhost` 是 Runner 自己，不是 Vercel。Vercel 随机链接用于追溯单次部署，验收和交付使用稳定生产别名 `https://mewmo.vercel.app`。

## 生产最低闸门

无论改动大小，生产 push 都要确认：改动相关测试通过、lint 通过、生产构建通过、工作区干净、本地与远端 `main` 同一提交、GitHub CI 成功、Vercel Production 达到 Ready，以及稳定生产别名冒烟通过。

最终报告要列明运行了什么、为什么相关、哪些检查未运行以及原因。没有对应新鲜证据时，不得声称 CI、Vercel Ready 或线上行为已验证。
