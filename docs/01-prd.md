## 1. 产品定位与差异化

### 一句话

mewmo 是一只住在你信息流里的 AI 猫——帮你收、帮你理、陪你想。

### 为什么现在做这个

信息管理赛道正在被两股力量夹击：

- **上方**：大厂系统级集成（Apple Intelligence、Google NotebookLM、腾讯 ima）免费 + 预装，把"收藏 + 摘要"降为平台功能
- **下方**：niche 工具极致垂直（Readwise 做高亮回顾、Cubox 做剪藏、Reor 做本地 RAG），每个做到 80 分

两股力量的共同盲区：**没人把"收集 → 消化 → 回溯"做成一个有情感、有陪伴感的完整体验**。大厂产品是功能堆叠，niche 工具是效率机器，都不提供"有人在帮我看着这些信息"的安心感。

### mewmo 的差异化叙事

不是更强的收藏夹，不是更智能的阅读器——是**一只认识你所有内容的猫**。它在你睡觉时帮你消化今天的订阅，在你搜索时从记忆里捞出相关的旧笔记，在你犹豫时说"这条和你上周看的那篇有关联"。

这不是 AI 外挂在工具上的"对话框"，而是产品本身就是围绕一只有记忆的猫来设计的：
- 猫"认识"你的内容（上下文注入不是可选功能，是默认行为）
- 猫的动作直接改变你的数据（打标、总结、归档——不是给你建议让你自己动手）
- 猫有自己的节奏（后台默默消化订阅、主动提醒遗忘内容）

## 2. 目标用户

| Persona | 核心场景 | 主要使用 | 为什么选 mewmo |
|---------|---------|---------|---------------|
| **信息饕餮者** | 每天 50+ 篇 RSS/长文，存了不看，看了记不住 | Feeds + Clips + AI 摘要 | 猫在后台替我消化，我只看摘要和猫的整理 |
| **笔记思考者** | 想法散落各处，想要一个打开就能写的地方 | Notes + 知识库 + 搜索 | 打开秒出内容，猫能帮我关联旧想法 |
| **Apple 重度用户** | Mac/iPhone 高频切换，要求菜单栏/快捷键/Widget | Apple 原生端全部功能 | 不是网页套壳，系统级集成 |

**共同画像**：每天高频打开（≥5 次）、快速记录、重视"打开瞬间就看到内容"。这类用户对延迟极其敏感——500ms 的等待就会让人放弃。

## 3. 设计原则

六条产品契约，按优先级排列。做产品决策时如有冲突，上面的赢。

1. **打开即看到** — 任何端启动先从本地缓存读取，不等网络。这是"有人在帮我看着"安心感的物理基础。
2. **猫是原住民** — AI 不是可选插件，是产品核心。猫的感知和动作贯穿所有功能。
3. **收集零摩擦** — 2 步以内把任何 URL 或文字落库。摩擦越低，用户越愿意把东西丢进来，猫能帮忙的越多。
4. **一个地方放一切** — 笔记/剪藏/订阅/对话共享同一个数据模型和搜索空间。不要变成又一个"分散在不同 Tab"的工具。
5. **云端真相、本地体验** — 数据在云端保证多端一致，体验在本地保证速度。冲突对用户无感。
6. **原生或不做** — Apple 端必须 SwiftUI 原生。如果做不到原生体验就先不做那个平台。

## 4. 用户旅程

### 4.1 新用户第一天

```
注册 → 看到空状态（猫打招呼，引导三条路）
         ├─ "粘贴一个 URL 试试" → 剪藏落库 → 猫说"我来读读"→ 自动摘要 + 打标
         ├─ "写点什么" → 新建笔记 → 写完后猫提示"要不要加个标签"
         └─ "订阅你常看的" → 添加 RSS → 猫说"我每小时帮你看一次新文章"
```

**成功标准**：第一次打开 5 分钟内完成至少一种收集动作。

### 4.2 日常使用循环

```
                    ┌─────────────────────────────────────────────┐
                    │                                             │
  收集 ──────→ 消化 ──────→ 回溯 ──────→ 产出（笔记/想法）──┘
  (剪藏/订阅/    (猫在后台      (搜索/浏览/
   快速记录)     摘要+打标)      猫主动关联)
```

**猫的角色在每个环节**：
- 收集时：确认"已收到"，告知会做什么
- 消化时：后台静默工作，完成后标记（用户下次打开看到结果）
- 回溯时：搜索时从全库召回相关内容，主动说"这和你之前看的 X 有关"

### 4.3 跨端场景

```
Mac 上看到文章 → 剪藏 → 关 Mac
                          ↓
iPhone 上打开 mewmo → 看到猫已经摘要好了 → 快速浏览 → 标"已读"
                          ↓
回到 Mac → 列表状态已同步 → 继续下一篇
```

## 5. 核心功能

### 5.1 笔记（Notes）

**用户价值**：打开就能写，写完自动存，随时找得到。

- Markdown 编辑 + 实时预览
- 标签、置顶、全文搜索
- 自动保存（用户无需手动点"保存"）
- 打开即渲染（从本地缓存读取，< 100ms）

**猫的参与**：可以请猫帮忙打标、扩写、总结、关联相关内容。

### 5.2 剪藏（Clips）

**用户价值**：看到好内容 2 步存下来，不需要再回去找。

- 浏览器扩展一键存 / 系统分享菜单 / 粘贴 URL
- 自动抓取正文并清洗（安全渲染，不执行外部脚本）
- 阅读态呈现（干净排版，无广告）

**猫的参与**：自动生成摘要 + 从用户标签池打标。用户下次打开看到"猫已读过"的标记。

### 5.3 RSS 订阅（Feeds）

**用户价值**：不用每天手动刷 RSS 阅读器，猫帮我看完告诉我哪些值得看。

- 添加源后自动每小时拉取
- 每条新文章自动生成 ≤240 字摘要
- 已读/未读跨端同步
- 支持按源、按标签过滤

**猫的参与**：自动摘要 + 打标是猫的"日常工作"，用户感知到"猫一直在帮我看"。

### 5.4 AI 猫咪对话（Cat Chat）

**用户价值**：一只认识我所有内容的猫，随时可以问它。

- 上下文注入当前打开的内容（笔记/剪藏/订阅文章）
- 可执行动作：搜索用户全库、打标签、生成摘要、关联内容
- 对话历史持久化，跨端可回顾
- 用户选择 AI 模型（Anthropic / OpenAI / DeepSeek）

**交互设计要点**：
- 猫是侧栏，不是全屏——它是"陪在旁边"，不是"抢走注意力"
- 猫的回答基于用户真实内容，禁止编造未读取的内容
- 猫的动作（打标/归档等）需先说明再执行，给用户确认感

### 5.5 搜索（Search）

**用户价值**：一个搜索框，搜到笔记/剪藏/订阅的所有东西。

- 全文关键词搜索（秒级返回）
- 语义搜索（embedding 向量检索，"关于 X 的内容"）
- 结果跨类型混合排序

### 5.6 增量同步（Sync）

**用户价值**：在哪写都行，到处都是最新的，不用想"我存在哪了"。

- Web ↔ Apple 双向同步
- 版本号驱动，冲突自动解决（不丢数据、不覆盖成空白）
- 删除软处理（跨端生效，可恢复）
- 同步在后台静默进行，用户无感

## 6. 猫的性格定义

猫不是通用 ChatBot，是有固定性格的伴侣角色。

| 维度 | 定义 |
|------|------|
| **语气** | 简短、温和、有点慵懒。不用感叹号，不过分热情，不说"很高兴为您服务" |
| **主动性** | 会主动做事（摘要/打标），但只在做完后轻声告知，不打扰 |
| **诚实** | 不知道就说不知道，不编造。"这个我没看过哦"比胡说八道好 |
| **记忆感** | 引用用户之前的内容时自然提起，让人感觉"它真的记得" |
| **边界** | 只做信息管理范围内的事。问它天气/写代码/闲聊，温和拒绝 |

## 7. 平台策略

```
第一批：Web + 浏览器扩展     ← 验证产品闭环（收集 → 消化 → 回溯一整条跑通）
第二批：Mac App             ← 核心用户群入口（菜单栏/全局快捷键/Spotlight）
第三批：iOS + iPad          ← 移动场景（分享扩展/Widget/Face ID）
```

**明确不做**：Windows · Android · 桌面 WebView 套壳

**原则**：Web 先行验证产品假设；Apple 原生用 SwiftUI 做体验差异化；不做的平台不降级做——宁可不做也不做成二等体验。

## 8. Scope 边界

### 2.0 必须有

- 笔记完整编辑体验（Markdown + 实时预览 + 自动保存 + 标签 + 搜索）
- 剪藏完整链路（收 → 存 → 读 → 猫摘要）
- RSS 订阅（添加 → 自动拉取 → 摘要 → 已读追踪）
- AI 猫咪对话（上下文注入 + 搜/标/总 动作）
- 增量同步基础（Web ↔ 未来 Apple 端）
- 浏览器扩展剪藏
- 深浅双主题
- 账号系统（邮箱注册/登录）

### 明确不做

- Windows / Android 客户端
- 协同编辑 / 多人实时
- 本地离线优先架构（产品是 cloud-first，本地只做缓存）
- 公开分享页作为核心功能（笔记分享是次级需求）
- 自研大模型（只接第三方 API）
- 社交/社区功能
- 付费/商业化（2.0 先做产品验证）

## 9. 成功指标

| 维度 | 指标 | 为什么这个重要 |
|------|------|---------------|
| **激活** | 新用户 5 分钟内完成首次收集动作 | 验证"零摩擦"承诺 |
| **留存** | 周活用户中有收集动作的比例 ≥ 60% | 证明不是"存了就忘" |
| **消化** | 剪藏/订阅条目被查看摘要的比例 | 证明猫的消化有人看 |
| **AI 渗透** | 周活用户中使用过猫对话的比例 | 证明猫不是摆设 |
| **体验** | 冷启动 < 500ms（原生端）/ 首屏即可见缓存（Web） | "打开即看到"的量化 |
| **同步** | 跨端同步成功率 ≈ 100%、冲突零丢数据 | 用户对"存在云端"的信任 |

## 10. 产品验收标准

从用户视角定义"做完了"——不是技术实现细节，是用户能感知的行为。

**打开体验**
- [ ] 打开 App/网页，内容瞬间可见（不是白屏等加载）
- [ ] 切换深色/浅色主题，所有界面可读、美观

**收集体验**
- [ ] 粘贴 URL → 2 步内生成剪藏，正文干净可读
- [ ] 浏览器扩展一键保存当前页面
- [ ] 新建笔记 → 即时可写，关掉再开不丢内容

**消化体验**
- [ ] 添加 RSS 源后 1 小时内出现新条目
- [ ] 每条新条目有摘要和标签（猫的自动工作结果可见）
- [ ] 剪藏被猫自动打标和摘要

**AI 猫咪体验**
- [ ] 在笔记/剪藏页唤出猫，猫能读到当前内容并基于它回答
- [ ] 请猫"搜 X"→ 返回用户库内真实结果
- [ ] 请猫"打标签"→ 标签真实写入，刷新可见
- [ ] 猫不编造内容，不知道就说不知道

**同步体验**
- [ ] 一端写了内容，另一端打开可见
- [ ] 两端同时改同一条，不丢数据

**安全与隐私**
- [ ] 剪藏渲染不执行外部脚本
- [ ] 用户只能看到自己的数据
- [ ] AI 调用走用户自己选的模型

## 11. 开放问题

| # | 问题 | 影响 | 当前状态 |
|---|------|------|---------|
| 1 | 猫的"主动提醒"能力边界 | 猫能不能主动推送"你有 3 天没看的内容了"？Push 还是只在打开时展示？ | 待定 |
| 2 | 知识库组织模式 | 笔记/剪藏之上的"文件夹/项目"层怎么设计？手动整理 vs 猫自动归类？ | 已有数据模型，产品交互待定 |
| 3 | 商业模式 | 免费额度 + 付费？猫的 AI 调用成本谁承担？ | 2.0 不考虑，但要预留接口 |
| 4 | 猫的记忆深度 | 猫能记住多久前的对话？能跨对话记住用户偏好吗？ | 技术上可行，产品定义待拍 |
| 5 | Apple 端 MVP scope | Mac 第一版做到哪些功能足够？全量还是裁剪？ | 待定（依赖 Web 验证结论） |

## 12. 竞争环境认知

不在 PRD 里做完整竞品分析（那是独立文档的事），但记录核心认知以锚定产品决策：

- **大厂免费化**：Apple Intelligence / ima / NotebookLM 正在把"AI + 收藏"降为平台基础功能。mewmo 不能和免费预装的东西比"有没有摘要功能"。
- **差异化在体验层不在功能层**：功能会被追平，但"一只有记忆的猫帮你管信息"的体验感受难以被复制——这需要产品每个细节都围绕猫来设计，不是加个 ChatBot 窗口。
- **赛道风险承认**：这是红海。mewmo 的赌注是"情感化 AI 伴侣 + 全链路闭环"能在功能趋同的市场里建立用户偏好。如果用户不为"猫"买单，产品假设失败。
