# ZOO-36 废纸篓详情体验设计

## 目标

废纸篓继续使用现有工作区双栏布局。左侧列表改为与“今天”一致的内容卡片，点击条目后在右侧查看被删除内容的只读详情。恢复和永久删除从列表卡片移到右侧详情顶部，避免管理按钮干扰内容浏览。

## 列表

`GET /api/trash` 只返回列表所需字段，不返回笔记或剪藏全文。列表项包含类型、标题、摘要、删除时间、剩余保留天数，以及数据本身已有的封面或站点图标。剪藏有 `coverImage` 时展示封面；其他类型没有可用封面时不制造占位图。

卡片复用“今天”的层级：标题在上，摘要为灰色辅助文字，底部依次显示类型图标、类型或来源、删除时间和剩余天数。整张卡片可点击并具有稳定的选中态。搜索结果变化、恢复或永久删除后，选择自动回落到下一条可见内容。

## 详情

在现有 `/api/trash/[kind]/[id]` 路由增加 `GET`。请求必须校验当前用户、内容类型和 `deletedAt`，只允许读取当前用户废纸篓中的记录。

详情按需加载：

- 笔记返回标题、摘要、Markdown 正文和时间信息，并以只读正文展示。
- 剪藏返回标题、摘要、来源、URL、封面和 HTML 正文，沿用现有安全清洗渲染链路。
- 订阅源返回标题、描述、URL、类型、站点图标和时间信息。
- 知识库返回标题、图标和时间信息。

右侧顶部显示“恢复”和“永久删除”。永久删除继续使用现有确认弹窗；恢复或删除成功后清理当前详情并选择下一项。详情加载失败时保留左侧列表，并在右侧显示局部错误状态。

没有废纸篓内容时，右侧显示空状态和“删除内容保留 14 天”的说明，不再显示类型统计面板。

## 数据边界

`createTrashRepository` 保留现有 `list`、`restore`、`deletePermanently`，新增按 `userId + kind + id + deletedAt` 查询的详情方法。列表查询为剪藏补充 `coverImage`、`favicon`、`excerpt` 等轻量字段，但不读取全文。详情查询按类型选择必要字段，不修改数据库结构。

## 测试

仓库测试覆盖详情查询的用户归属和已删除约束，以及各类型返回字段。路由测试覆盖 `GET` 的认证、类型校验和 404。页面测试覆盖可选列表卡片、封面、按需详情请求、四种详情渲染、右侧恢复与永久删除，以及操作后的选择回落。完成后运行全量测试、lint、主题色检查和构建，并在桌面与移动视口进行浏览器截图检查。
