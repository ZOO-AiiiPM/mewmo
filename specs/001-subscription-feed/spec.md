# Feature Specification: 订阅区（Subscription Feed Zone）

**Feature Branch**: `001-subscription-feed`
**Created**: 2026-05-21
**Status**: Ready for Plan
**Input**: User description: "订阅功能（subscription / feed）：vibe coding 桌面 app 新增第四个 zone（已在架构里预留 Zone='subscribe'），让用户订阅外部内容源（RSS / Atom 等）并在订阅区查看更新。"

## User Scenarios & Testing *(mandatory)*

<!--
  User stories 按优先级 P1 / P2 / P3 组织，每条独立可测、独立交付价值。
  P1 = 没它整个功能不成立的核心闭环；P2 = 价值放大；P3 = 长期使用必备但非首发。
-->

### User Story 1 - 添加订阅源并看到更新内容（Priority: P1）

用户进入订阅区，提供一个外部内容源的标识符（URL 或类似），把它加入到自己的订阅列表中。系统能从该源获取内容，并在订阅区按时间倒序展示该源的最新条目（标题、摘要、发布时间、来源）。

**Why this priority**: 这是整个订阅功能的最小闭环——"订阅 → 看到更新"。没有它，订阅区就是空架子；有了它，用户已经能用这个区做日常的"信息捕获"。所有 P2/P3 故事都建立在这一条之上。

**Independent Test**: 给一个公开可达的内容源标识符 → 用户在订阅区添加它 → 在合理时间内（< 5 分钟）能看到来自该源的至少一条内容。

**Acceptance Scenarios**:

1. **Given** 用户首次进入订阅区，**When** 提供一个有效的源标识符并提交，**Then** 该源出现在订阅源列表中，并开始抓取内容
2. **Given** 已添加的源 X 有新发布的内容，**When** 用户回到订阅区，**Then** 看到来自 X 的新内容条目（含标题、摘要、发布时间、来源）按时间倒序排列
3. **Given** 提供的源标识符无效或源不可达，**When** 用户提交，**Then** 看到清晰的错误状态（说明问题所在），不静默失败也不崩溃
4. **Given** 源暂时不可达后恢复，**When** 用户重新触发抓取或下次定期抓取，**Then** 内容能正常出现，源状态恢复正常

---

### User Story 2 - 阅读单篇内容的完整正文（Priority: P2）

用户从订阅区的内容列表里点选一条，进入详情视图阅读完整正文（含图片、链接、格式化文本）。阅读体验要在亮色 / 暗色主题下都清晰可读。

**Why this priority**: 标题列表只能让用户"知道有什么"，不能让用户"消费内容"。详情阅读是订阅区从"信息流入口"变成"实际可用阅读工具"的关键步。

**Independent Test**: 在 P1 已有内容的基础上，点击任意一条进入详情，确认能完整看到正文内容、图片正常加载、暗色模式不出现"白底色块"或"文字看不见"等已知问题。

**Acceptance Scenarios**:

1. **Given** 列表中存在一条内容条目，**When** 用户点击它，**Then** 视图切换到该条目的详情阅读页，显示完整正文
2. **Given** 内容含内嵌图片，**When** 详情视图渲染，**Then** 图片正常加载且布局协调
3. **Given** 用户当前为暗色主题，**When** 阅读源中含 inline 颜色（如 `<span style="color:rgb(63,63,63)">`）的内容，**Then** 灰阶文字适配主题色（不出现"白底深字看不见"），彩色装饰保留作者意图
4. **Given** 用户回到列表，**When** 同一条目再次出现，**Then** 已读状态被记住

---

### User Story 3 - 管理订阅源（Priority: P3）

用户能查看所有已订阅的源、删除不再需要的源、看到每个源的状态（最近抓取时间、最近抓取是否成功）。

**Why this priority**: 长期使用后必然产生"订阅积累 → 清理"需求，但首次发布前用户的订阅数量很少，管理压力不强。先做 P1+P2 验证核心价值，P3 在用户实际订阅累积后再补足。

**Independent Test**: 添加若干个源后进入源管理界面，能看到全部、能删除任意一个、删除后该源及其内容条目不再出现在列表中。

**Acceptance Scenarios**:

1. **Given** 用户已添加多个订阅源，**When** 进入源管理视图，**Then** 看到所有源及其元信息（名称、URL、最近抓取时间、状态）
2. **Given** 用户删除某个源，**When** 确认删除，**Then** 该源从列表消失，相关历史内容也不再出现
3. **Given** 某个源最近一次抓取失败，**When** 用户查看源管理视图，**Then** 看到错误状态及时间戳

---

### Edge Cases

- 源标识符格式错误（不是合法 URL / 不是支持的协议） → 添加时立刻校验并报错
- 源暂时不可达 vs 永久 404：暂时不可达保留源 + 标记错误状态，用户后续可重试；永久 404 仍保留源（避免自动删除导致历史内容丢失），由用户决定是否删除
- 单次抓取返回大量历史内容（如新订阅一个有上千条历史的源）：系统应能稳定处理，不阻塞 UI
- 同一篇内容在不同源出现：作为不同条目分别展示（不做跨源去重），避免误判
- 订阅源数量增长到几十个甚至上百个：列表性能不应明显下降
- 内容含恶意 / 越界格式（畸形 HTML、超大图片等）：阅读视图应具备容错，不崩溃整个 app
- 用户离线状态：已抓取的内容仍可阅读；新抓取请求应明示"网络不可用"

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 用户必须能够通过提供源标识符添加新的订阅源
- **FR-002**: 系统必须能够从已添加的订阅源获取内容（首次添加时获取历史；之后定期或按用户触发获取增量）
- **FR-003**: 系统直接支持的源类型为 **RSS 2.0 和 Atom 1.0** 标准协议的源。非标准平台（公众号 / X / YouTube / Substack newsletter 等）通过**用户自部署的第三方桥接服务**（推荐 we-mp-rss，备选 wewe-rss）输出 RSS URL 后再接入——vibe-coding 不下场爬这些平台
- **FR-003a**: vibe-coding **不内嵌、不打包、不分发**任何爬虫或桥接服务代码——避免 license 传染（AGPL）和反爬合规风险。文档中给出"如何 setup 桥接服务"的 onboarding 指南，但服务本身由用户自负责
- **FR-004**: 用户必须能够查看所有已订阅的源的列表
- **FR-005**: 用户必须能够查看每个订阅源的内容流（按发布时间倒序）
- **FR-006**: 用户必须能够进入单条内容的详情阅读视图，看到完整正文
- **FR-007**: 用户必须能够删除已订阅的源（包括其历史内容）
- **FR-008**: 系统必须在源不可达时显示清晰的错误状态（区分网络问题 vs 源本身错误）
- **FR-009**: 系统必须将抓取到的内容持久化保存，支持离线阅读
- **FR-010**: 系统必须记忆每条内容的已读 / 未读状态
- **FR-011**: 订阅区的视觉语言必须与已成熟的笔记区保持一致：左列表（按时间分桶）+ 右详情的双栏布局、light/dark 主题适配模式、Zone 切换交互
- **FR-012**: 订阅区必须接入既有的 Zone 切换状态机（已预留 `'subscribe'` zone 入口）
- **FR-013**: 本期 spec **不包含** Daily Brief（跨源聚合 + AI 摘要日报）。Daily Brief 属于 AI 加工层功能，与"AI 织网"等其它 AI 加工能力一起，由独立 spec 处理；本 spec 仅完成"订阅 + 抓取 + 阅读"的基础闭环

### Key Entities *(include if feature involves data)*

- **订阅源（Subscription Source）**：用户主动添加并希望持续接收更新的外部内容来源。属性包括标识符（URL）、显示名、添加时间、上次抓取时间、上次抓取状态（成功 / 失败 / 进行中）、状态描述
- **内容条目（Feed Entry）**：从订阅源抓取的单条内容。属性包括标题、正文、发布时间、来源源（关联订阅源）、抓取时间、原文链接、已读状态、可选摘要 / 缩略图

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从进入订阅区到完成添加第一个源的全流程，在 60 秒内可完成
- **SC-002**: 添加一个有效源后，5 分钟内能看到该源的至少一条历史或最新内容（在网络畅通条件下）
- **SC-003**: 对常见的公开可访问标准订阅源（指业界主流格式的内容源），首次抓取成功率 ≥ 90%
- **SC-004**: 订阅区与笔记区在布局结构、间距、字号、主题适配等维度的视觉一致性评估通过——用户在两个区切换不感到"风格断裂"
- **SC-005**: 单篇内容的阅读体验（图片显示、暗色模式适配）与既有的剪藏阅读器持平，不出现"白底深字看不见"或"图片不加载"等回归问题
- **SC-006**: 在订阅源数量达到 50 个、内容条目总数达到 5000 条时，订阅区列表加载和滚动仍保持流畅（无明显卡顿）
- **SC-007**: 当源不可达时，用户能在错误发生后 3 秒内看到清晰的错误状态指示，不出现静默失败

## Assumptions

- **版本归属**（用户决策 2026-05-22）：本功能纳入 **v1（当前版本）**。事实核查后澄清：宪法 v2.0.0 正文 **不含**版本路线图（5 条核心原则 + Product Scope 都未规定 RSS 必须 v2），因此**无需修宪**。需要的是更新 README.md 的进度标记（RSS 多源捕获从 P1.2 待开始 → P1.x 进行中），并在 journal 记录路线图调整。Daily Brief 等 AI 加工层功能维持后续版本
- **数据存储**: 复用项目既有的本地数据存储机制，不引入新的存储层
- **UI 视觉系统**: 复用项目既有的主题系统（light/dark 成对定义）和布局模式（List + Detail 双栏），不新增独立设计语言
- **Zone 集成**: 架构层面 `Zone = 'subscribe' | 'notes' | 'clipping' | 'sediment'` 已预留 `'subscribe'`，本功能直接占用该入口
- **与剪藏的边界**: 剪藏（clip）= 用户被动单次保存某个网页；订阅（subscription）= 用户主动持续接收某个源的更新。两者都是"外部内容捕获"，但触发机制和生命周期不同，应作为并列功能区而非合并
- **AI 织网集成范围**: 本期 spec 仅完成核心 loop 中的"捕获"环节（订阅 + 抓取 + 展示）。后续"整理 → 激活 → 消费 → 沉淀"环节（如 AI 摘要、跨内容关联、智能推送等）由后续 spec 处理，不在本期范围
- **抓取触发策略**（用户决策 2026-05-22）：每天 1 次 batch 抓取；触发时机 = app 启动时检查"今天没抓过则抓一次" + 用户手动点"刷新"按钮；app 关闭后不抓（不接 OS 级后台任务）。覆盖了"信息天级新鲜度"的常规场景，紧急更新由用户手动刷新或去原网站。
- **多语言**: 默认假设支持 UTF-8 文本和图片内容；不专门处理多语言搜索 / 翻译等功能
- **同步**: 本功能不涉及多设备订阅同步，仅在本机维护订阅列表和内容缓存
