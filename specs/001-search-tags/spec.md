# Feature Specification: 全局搜索 + 标签管理

**Feature Branch**: `001-search-tags`
**Created**: 2026-05-22
**Status**: Draft
**Input**: User description: "为 vibe-coding 笔记 / 剪藏 app 加全局搜索 + 标签管理功能模块（一起设计：标签 schema 早一步加可以避免 FTS5 后续加列重建索引）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 全局关键词搜索 (Priority: P1)

用户在搜索框输入中文 / 英文 / 中英混合关键词，立即看到所有命中的笔记和剪藏，按相关度（标题命中优先 > 标签命中 > 正文命中）+ 时间新鲜度排序，分组展示「笔记 X 条 / 剪藏 Y 条」。

**Why this priority**: 这是搜索功能的最小可用切片。没有它整个功能模块没有用户价值。"找回过去记的某条笔记 / 某个剪藏"是日常最高频场景。

**Independent Test**: 在已有 ≥10 条笔记和 ≥5 条剪藏的应用中，输入一个出现在某条笔记标题里的中文关键词，验证该笔记排在结果第 1 位 + 命中关键词视觉高亮 + 笔记 / 剪藏分组计数正确。

**Acceptance Scenarios**:

1. **Given** 用户已有 5 条笔记，其中标题"机器学习入门"含目标关键词，**When** 在搜索框输入"机器学习"，**Then** 该笔记出现在结果列表第 1 位
2. **Given** 用户的某条笔记正文里提到"读书心得"但标题是"周末"，**When** 搜索"读书"，**Then** 该笔记出现在结果中（排序在标题命中之后）
3. **Given** 用户搜索"AI"（≤2 字短词），**When** 提交，**Then** 包含"AI"的笔记被正确命中（不退化为不相关 noise）
4. **Given** 用户同时有笔记和剪藏命中关键词，**When** 搜索，**Then** 结果按"笔记 X 条 / 剪藏 Y 条"分组展示
5. **Given** 用户搜索一个不存在的词，**When** 搜索，**Then** 显示"无结果"提示，不卡死、不报错
6. **Given** 输入了多个关键词（"机器学习 项目"），**When** 搜索，**Then** 同时含两个词的笔记排前

---

### User Story 2 - 标签创建、关联与浏览 (Priority: P2)

用户能给笔记 / 剪藏挂多个标签（输入 `#` 触发自动补全），能浏览所有同标签下的内容。标签改名一处生效，删除标签不会级联删除关联的笔记 / 剪藏。

**Why this priority**: 标签是组织 / 二级分类能力，是搜索之外的第二条找回路径。但部分用户"全文搜索就够用"——所以排 P2 而非 P1。

**Independent Test**: 给一条笔记加 3 个标签 → 在标签列表里看到这 3 个标签 → 点击其中一个 → 进入该标签下所有笔记 + 剪藏视图，看到那条笔记。

**Acceptance Scenarios**:

1. **Given** 用户在编辑笔记时输入 `#`，**When** 继续输入字符，**Then** 立即看到已有标签的自动补全下拉
2. **Given** 用户选中一个已有标签，**When** 确认，**Then** 笔记保存后该标签关联到笔记
3. **Given** 用户输入新标签名提交，**When** 确认，**Then** 系统创建新标签并关联到当前笔记
4. **Given** 应用界面有标签列表 / 云，**When** 点击某标签，**Then** 进入"该标签下所有笔记 + 剪藏"视图
5. **Given** 用户改了一个标签的名字，**When** 改完，**Then** 所有关联的内容显示新名（一处改全局生效）
6. **Given** 用户删除一个标签，**When** 确认，**Then** 标签消失，但原来的笔记 / 剪藏保留（只失去这个标签）

---

### User Story 3 - 搜索结果质感与扩展能力 (Priority: P3)

用户在搜索结果里直观看到"为什么命中"——命中关键词高亮 + 约 32 字上下文片段 + 元数据搜索 + 前缀通配。

**Why this priority**: 搜索体验质感的"提升项"——P1/P2 跑通后才考虑。让"找得到"升级为"找得舒服"。

**Independent Test**: 搜索一个有多条命中的关键词，验证每条结果有 32 字上下文 + 命中词视觉高亮 + 标题命中排前 + 时间衰减生效。搜索剪藏的站点名也能找到相应剪藏。

**Acceptance Scenarios**:

1. **Given** 用户搜索"机器学习"命中多条结果，**When** 浏览结果列表，**Then** 每条结果展示约 32 字上下文片段，关键词视觉高亮
2. **Given** 一条笔记标题命中、另一条仅正文命中，**When** 排序，**Then** 标题命中的笔记排在更前
3. **Given** 两条笔记相关度相近，但一条更近改过，**When** 排序，**Then** 最近改过的排前
4. **Given** 用户搜索"少数派"，**When** 提交，**Then** 来自该站点的所有剪藏出现在结果中
5. **Given** 用户搜索"深度学*"，**When** 提交，**Then** 命中"深度学习"、"深度学派"等以"深度学"开头的内容

---

### Edge Cases

- 空搜索（输入纯空格 / 0 字符）→ 不报错，提示"输入关键词"
- 笔记 / 剪藏数据库为空时搜索 → 显示"还没有内容"，不显示无结果错误
- 搜索词含特殊字符（标点 / emoji）→ 系统忽略不可索引字符，正常返回结果
- 搜索结果超过 1000 条 → 截断到前 50 条 + 提示"更多结果请缩小搜索词"
- 标签名重复（用户尝试创建已存在标签）→ 复用已有标签，不报错
- 删除标签时该标签下还有内容 → 解除关联（内容保留），不级联删除
- 笔记 / 剪藏被删除 → 标签关联自动失效，不留 dangling 引用
- 极长搜索词（超过 100 字）→ 截断或提示输入过长

## Requirements *(mandatory)*

### Functional Requirements

#### 搜索基础

- **FR-001**: 系统必须支持对笔记和剪藏的关键词全文搜索
- **FR-002**: 用户必须能搜索到 ≥3 字的完整中文词（如"机器学习"、"产品设计"）
- **FR-003**: 用户必须能搜索到 ≤2 字的中文短词（如"AI"、"日本"、"会议"）—— 不能因短词长度退化精度
- **FR-004**: 用户必须能搜索中英文混合词（如"ChatGPT 笔记"、"iOS 体验"）
- **FR-005**: 系统必须按相关度排序，标题命中权重 > 标签命中权重 > 正文命中权重
- **FR-006**: 系统必须按时间新鲜度加权（相关度相近时新的排前）
- **FR-007**: 搜索结果必须按内容类型分组展示（"笔记 X 条 / 剪藏 Y 条"）
- **FR-008**: 系统必须高亮命中关键词在结果中的位置
- **FR-009**: 系统必须返回带上下文的约 32 字符片段
- **FR-010**: 用户必须能搜索到剪藏的元数据（站点名、作者）
- **FR-011**: 系统必须支持前缀通配（如"深度学*"匹配"深度学习"、"深度学派"）
- **FR-012**: 多关键词输入时必须按 AND 处理（同时含所有词的内容排前）

#### 标签管理

- **FR-013**: 每条笔记 / 剪藏必须能挂多个标签（多对多关系）
- **FR-014**: 用户必须能通过 `#` 触发标签输入补全器
- **FR-015**: 标签输入器必须显示已有标签的实时自动补全
- **FR-016**: 用户输入新标签名必须能直接创建并关联
- **FR-017**: 系统必须提供标签列表 / 云的浏览入口
- **FR-018**: 点击标签必须进入"该标签下所有内容"视图
- **FR-019**: 标签改名必须一处改全局生效（所有关联内容显示新名）
- **FR-020**: 删除标签必须不级联删除关联的笔记 / 剪藏
- **FR-021**: 在搜索框输入标签名时，系统必须能找到挂了该标签的内容（标签纳入搜索索引）

#### 性能与索引

- **FR-022**: 在 1000 条笔记 + 500 条剪藏数据规模下，99% 搜索请求必须在 100 ms 内返回结果
- **FR-023**: 用户修改笔记 / 剪藏内容后，搜索索引必须实时同步（改完后立即可搜到新内容）

#### 边界处理

- **FR-024**: 系统必须处理空搜索 / 纯空格输入（不报错，提示输入关键词）
- **FR-025**: 笔记 / 剪藏被删除后，相关标签关联必须自动失效

### Key Entities

- **笔记 (Note)**: 用户创作的内容，含标题、正文（Markdown）、创建 / 更新时间
- **剪藏 (Clip)**: 来自外部链接的内容，含 URL、标题、正文、摘要、站点名、作者、保存时间、封面图
- **标签 (Tag)**: 用户定义的分类标记，名称唯一，1-50 字符
- **标签关联 (Note-Tag / Clip-Tag)**: 笔记 / 剪藏 与 标签 的多对多关系，删除任一端时自动断开关联

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ≤2 字中文短词搜索（"AI"、"日本"、"会议"）的召回率 ≥ 95%（已有内容里含该词的条目，95% 以上能被找到）
- **SC-002**: ≥3 字中文完整词搜索（"机器学习"、"产品设计"）的召回率 100%
- **SC-003**: 1000 笔记 + 500 剪藏的数据规模下，99% 搜索请求让用户在 100 ms 内看到结果
- **SC-004**: 用户从输入第一个字符到看到结果的端到端感知延迟 ≤ 500 ms（含输入防抖与渲染）
- **SC-005**: 标题里含搜索词的条目排序位置 100% 优于仅正文里含的条目
- **SC-006**: 用户在 30 秒内完成"创建新标签 → 关联到一条笔记 → 浏览该标签下所有内容"的端到端任务，完成率 ≥ 95%
- **SC-007**: 标签名长度限制 1-50 字符（避免极端长度造成 UI 异常）
- **SC-008**: 每条笔记 / 剪藏支持挂载至少 20 个标签

## Assumptions

- 用户主要使用中文输入笔记和剪藏，英文为辅助场景（不为纯英文用户优化）
- 单机本地使用，不考虑多端同步与协作冲突（v1 范围）
- 数据规模假设：万级以内笔记 + 千级以内剪藏 / 标签百级
- 现有 zone 结构（笔记 / 剪藏 / 订阅 / 沉淀）保持不变；订阅 / 沉淀本身未实现，搜索暂只覆盖笔记 + 剪藏
- 标签由用户自主定义和管理，系统不预设标签库或推荐
- 搜索入口：复用 Sidebar 顶部既有搜索框（之前是占位 input，本次接入实际逻辑）
- 新词 / 网络流行词（"vibe coding"、"Claude Code"）的搜索精度依赖底层中文分词词典覆盖度，超出词典的边缘词由 fallback 机制兜底，不保证 100% 命中
- 暂不支持搜索的复杂布尔语法（AND / OR / NOT / 括号），多关键词默认按 AND 处理
- 暂不考虑拼音、错字、同义词、语义查询、图片 OCR，作为 P2+ 增强项另起评估
