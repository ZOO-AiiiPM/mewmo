# 日期分组用 rolling window，不要用日历边界

## 现象

给笔记 App 加日期分组（今天 / 昨天 / 本周 / 本月 / 本年 / 更早），第一版下意识按日历边界写：「本周 = 本周一至今」「本月 = 本月 1 号至今」。代码看起来很合理：

```ts
const dayOfWeek = (today.getDay() + 6) % 7; // 周一 = 0
const weekStart = new Date(today);
weekStart.setDate(today.getDate() - dayOfWeek);
const monthStart = new Date(year, month, 1);
```

5/21 周四上线。用户立刻反馈：「5/16（5 天前）的笔记怎么跑到『本月』里了？明明是本周啊。」

## 根因：日历边界对周初/月初严重反直觉

5/21 周四的"本周一" = 5/18，5/16 周六比 5/18 还早，按日历定义就是"上周"，于是被归到「本月」桶。

但中文口语「本周」≠「本周一至今」，更接近「距今一周内」。「本月」也偏向「过去 30 天」而非「本月 1 号至今」。日历边界还会在每周一 / 每月 1 号制造诡异跳跃 —— "昨天这条还在本周，今天怎么跳到本月了？"用户感知是 bug。

## 修复：4 行 rolling window 覆盖所有边界

```ts
const dayMs = 86_400_000;
if (d >= today.getTime() - 7   * dayMs) return 'week';
if (d >= today.getTime() - 30  * dayMs) return 'month';
if (d >= today.getTime() - 365 * dayMs) return 'year';
```

没有"周一是周几"、"这个月几号"这类锚点逻辑，没有跨周/跨月跳变。

## 蒸馏 1：能用一句话讲给非技术用户听吗？

- rolling window：「过去 7 天 / 30 天 / 365 天 / 更早」—— 任何人秒懂
- 日历边界：「本周一至今 / 本月 1 号至今 / 本年 1/1 至今」—— 需要解释，且每周一/每月 1 号会出现"昨天还在本周，今天到本月了"的跳跃

分组语义如果连一句话都解释不清楚，UI 上就一定会被用户当成 bug。

## 蒸馏 2：什么时候反而该用日历边界

当语义本身就是日历的 —— "本月支出统计"、"本年税务汇总"、"本周销售目标"。这些场景用户**主动按月/周做账**，日历边界才是对的。

判定标准：
- 分组是为了**回看 / 检索**（最近写了啥）→ rolling 赢
- 分组是为了**对账 / 周期对齐**（这个月花了多少）→ calendar 赢

## 反直觉处

日历边界看起来"更精确"、更符合编程直觉 —— 有明确锚点（周一 / 1 号 / 1/1），可计算、可测试、边界清晰。但**精确不等于正确**：用户脑子里的"本周本月"本来就是模糊的、滑动的。

反过来推：**用户用模糊语言描述需求时，往往他真正要的是模糊语义**（rolling），而不是精确语义（calendar）。开发翻译用户语言时，rolling-by-default 是更安全的猜测；只有当用户明确提到"账期"、"月底"、"周报"这类周期词时，才该切回日历边界。
